# cascade-img — Lead Review Report

## 1. Overall verdict

cascade-img is, on substance, a stronger v0.1 than most pre-1.0 OSS projects: the engineering is careful (honest sync/async boundary, structured `{ok, result | error:{code,message,remediation}}` envelopes, a thoughtful bridge-resilience layer), the numbers in the user-facing docs are accurate (113 tests, 36 tags, 10 MCP tools all verify clean at HEAD), and OPERATIONS.md is a genuinely excellent runbook. **The biggest strength is operational honesty and rigor — failure-mode catalogs, a real CHANGELOG, a CI matrix, and a worked example that is careful to label itself as not-a-template.** The biggest liabilities are three: (a) a real, untested CLI bug — `cascade-mj`'s only live code path awaits synchronous methods and raises `TypeError`; (b) the published-package experience is broken at two points — the first `cp` command in the human quickstart targets a file that does not ship in the wheel, and the PyPI long-description is a stale "in progress" placeholder; and (c) the repo ships ~65 files of builder-only process material (with leaked local paths, a live Discord CDN URL, and a real name) that no consumer needs and that has already leaked staleness back into a consumer-facing file. Against popular-OSS norms the project is **about right on content, too sparse on standard community files (no SECURITY/CoC/templates/badges), and mildly over-written in register** — the opposite of the usual OSS failure mode.

## 2. Writing quality (Orwell / Strunk & White)

The prose splits cleanly. The **operator-manual register** (OPERATIONS.md failure-mode catalog, AGENTS.md loop diagram and field tables, vocabulary/README.md) is plain, active, concrete — model technical writing. The **marketing-facing surfaces** (README lead, "What makes this different," the structured-events paragraph) and the **closing aphorisms** break the high bar in specific, fixable ways: a 78-word lead sentence, a parade of unglossed jargon, a dead-metaphor pile, and two empty maxims. ### On "V7 facets"

**Plain-English gloss (add on first use):** "Midjourney is normally driven by one long text string. cascade-img splits that string into separate, reusable parts — what to draw, which style to copy, which character to stay faithful to, and how strongly — so an agent can change one part without rewriting the rest."

**Is "facets" the right word? No.** It is a fancy word (Strunk: avoid fancy words; Orwell ii) carrying no precise technical meaning here, and it is not what Midjourney, the SDKs, or users call these. It also appears at README:3, README:69, and AGENTS.md:54 and is never glossed before use.

Ranked alternatives:

| rank | term | rationale |
|---|---|---|
| 1 (recommend) | **prompt parts** / "composable prompt parts" | Plainest; says exactly what they are. Zero glossary needed. |
| 2 | reference inputs | Accurate for `--p`/`--sref`/`--oref`, but `--ow`/`--s` are weights, not references — under-covers. |
| 3 | parameters | Correct and familiar, but generic; loses the "stack independently" idea that is the selling point. |
| 4 | controls / knobs | Clear but informal for a README headline; "controls" implies runtime UI. |
| 5 | modifiers | MJ-community-adjacent but ambiguous (the subject text is not a modifier). |

**Single recommendation:** lead with **"composable prompt parts"** and, on first mention only, name them in MJ's own terms in parentheses. Keep "facets" out of the public docs entirely. Also retire the related undefined jargon: "first-class inputs" → "set each part independently"; "without a human in the room for every roll" → "with no human needed on every attempt" (and use one phrasing consistently — README:3 and AGENTS.md:9/29 currently vary it).

## 6. Missing features — v0.2+ candidates (implementation-grade)

The HEAD surface is deliberately narrow: an eight-flag composer, a two-interaction bridge (`/imagine` + U1–U4), and a three-step curation chain (crop → key → promote) where selection is a bare quadrant int. The candidates below deepen each of those surfaces. Every one was verified against current source line-by-line and against current (2026) Midjourney references; one specialist proposal (multi-prompt `::` weighting) was dropped after verification showed the syntax is inert on `--v 7`, which the composer hard-codes at `composer.py:100`.

Two cross-cutting facts shape this whole section and are worth stating once:

- **The routing token is the bridge's spine.** Every job carries a per-job `request_token` woven into the prompt as `--no cscidnocollide{token}` (`Job.tagged_prompt`, `bridge.py:278-282`); the three matchers key on that substring (`_token_needle` returns at `bridge.py:474`; `_match_grid` substring checks at `:491`/`:498`; `_match_upscale` at `:516`). Any composer-emitted `--no` must **merge** into that one clause, not produce a second `--no`. And every *derived* MJ action (V1–V4, Reroll, Zoom, Pan) re-echoes the **parent** prompt — and therefore the **parent** token — so token routing is actively wrong for child grids and a new parent-uuid/nonce match path is mandatory.
- **The vocabulary is a locked catalog** (`versions/0.1.json:3 "locked": true`) validated at emit time. New tags must be added to that file or `emit()` raises (`_runtime.py:65-69`). Note one nuance the specs sometimes overstated: `validate()` rejects *unknown tags* and *missing required fields* only — it does **not** reject *extra* fields (`_runtime.py:70-76`). Adding a new tag is a deliberate edit that must keep the vocab-tag/emit-callsite parity gate green (`tools/check_vocabulary_parity.py`, currently 30 tags / 40 callsites).

### Top bets

A prioritized shortlist by value-to-effort ratio. The owner's two flagged curation gaps — **selection** and **alpha-keying recovery** — head the list.

1. **ML background-removal fallback (`rembg`)** — the single highest-leverage curation upgrade and the owner's flagged alpha-keying gap. Turns a guaranteed reroll on gradient/vignette/textured backgrounds into a salvage. Ships behind an optional `[ml]` extra; pure-PIL stays the default. *Effort: M.*
2. **Automated candidate scoring (`score_grid`)** — the owner's #1 flagged gap (selection). Replaces the blind quadrant int with a numeric ranking the agent picks on evidence and logs. Pure-PIL default; the `numpy`/`cv2` path is deferrable. *Effort: M.*
3. **Negative prompt (`--no`)** — the most-requested locked-style control (suppress text, watermarks, limbs), and the only one that touches the routing token. Cheap once the token-merge seam is right. *Effort: M.*
4. **V1–V4 variations** — the missing middle of the curation loop. Reuses the exact `_press_button` seam; the real work is the new child-job/derived-match machinery that several other interactions then inherit for free. *Effort: M.*
5. **Persistent job store (SQLite)** — the biggest production footnote: a restart silently drops every in-flight job while MJ keeps billing. Stdlib `sqlite3`, write-through on `Job.touch()`. *Effort: M.*
6. **Core composer params (`--chaos`, `--seed`, `--q`, `--stop`, `--weird`)** — five small, independent, dependency-free flags that together unlock exploration, reproducibility, and cost control. Each is S; bundle them. *Effort: S each.*

Honest caveats to keep in view throughout: the `--no` token merge must keep the token clause terminal or matching breaks; `--tile` likely conflicts with the U1–U4 upscale path on v7; `score_grid`'s accuracy/`cv2` trade-off; and Vary (Region) remains the one honest defer because its mask encoding is editor-centric.

#### Negative prompt (`--no`), merged with the bridge routing token

- **Gap:** `composer.py` emits no `--no` at all (flag block 98–122). The only `--no` on the wire is bridge-injected (`Job.tagged_prompt`, `bridge.py:282`). If the composer also emitted `--no text`, the wire prompt would carry two `--no` flags; MJ wants one comma-list, and a naive merge could drop the token, breaking `_match_grid`/`_match_upscale` (`:491`/`:498`/`:516`).
- **Surface:** `Subject.negatives: list[str] = field(default_factory=list)` (negatives are about the subject, not the style). `compose()` emits exactly one clause when non-empty: `--no ` + `", ".join(stripped non-empty negatives)`, appended **last** among non-routing flags. `facets_used` gains `"negatives"`. `mcp_server.compose_prompt` gains `negatives: list[str] | None = None`, threaded into `Subject(..., negatives=negatives or [])`.
- **Implementation:** Add `negatives` to `Subject` after `constraints` (`composer.py:27`). In `compose()`, after the identity block (~line 122), append the single `--no` clause. The merge is owned by the bridge: extract a module-level `_merge_no_clause(prompt: str, token: str) -> str` and call it from `tagged_prompt()` (replacing `bridge.py:282`). It finds a trailing `--no <list>` via an end-anchored, non-greedy regex `r'--no\s+(.+?)\s*$'`; if present it returns `prompt[:start] + '--no ' + existing_list + ', ' + token`, else it returns today's `f'{prompt} --no {token}'`. `_token_needle` is unchanged because `cscidnocollide{token}` still appears verbatim as a substring (the matchers do not require it to be its own flag).
- **Caveats:** MJ moderation reads each `--no` word independently, so multi-word negatives must be comma-separated, not space-joined — the composer's comma-join is correct. The composer's `--no` must always be emitted **last** and the bridge regex anchored to `$`, so the token stays the terminal comma element and MJ echoes `cscidnocollide{token}` contiguously. Today the bridge drives exactly two MJ interactions: a type-2 slash command (`_send_imagine`/`_post_interaction`, `bridge.py:842`/`819`) and a type-3 component press for U1–U4 (`_press_button`/`_press_all_slots`, `bridge.py:880`/`662`). Two shared facts drive the whole bucket: (a) the bridge currently acts only on the grid it just matched inside `_ingest_message`, and never stores the **upscaled single-image** message's `message_id` or `.components` (the upscale-complete handler at `bridge.py:747-794` downloads and completes, then discards components) — so Vary/Zoom/Pan/Make-Square buttons are unreachable until that message is persisted; and (b) every derived action re-echoes the **parent** prompt + token, so token routing is wrong for child grids and a **new parent-uuid/nonce-keyed derived-match path** must be built. The MJ button/command set is confirmed live in 2026 docs; the `MJ::JOB::` custom_id family is confirmed via the erictik/midjourney-api source.

### Curation & composition

The curation chain is a thin three-step spine with no intelligence at the seams: `crop_quadrant(src, quadrant)` (`crop_grid.py:30`, fixed 2×2 split keyed off `QUADRANT_OFFSETS` at `:22`, confirmed to match MJ's U1=TL/U2=TR/U3=BL/U4=BR layout), `alpha_key_corners(img, tolerance, method)` (`alpha_key.py:175`, flood/threshold, emits `ALPHA_KEY_APPLIED` with `keyed_count`/`total_count`; **returns `Image.Image` only, not a tuple**), and `promote(src, dest)` (`promote.py:15`, single-file byte copy at `:33-34`). The three flagged gaps: **selection** is a blind quadrant int; **alpha-keying** has two pure-PIL algorithms and no recovery; **promote** is byte-copy with no packing/format/dedup. Every feature below slots into crop → key → promote, and every new signal must be added to `versions/0.1.json` under the **CurationKit** emitter (`:64`) and keep the parity gate (`tools/check_vocabulary_parity.py`) green; new MCP tools must be added to the tool-enum at `versions/0.1.json:74` in lockstep.

#### ML background removal (`rembg`) as an alpha-key fallback

- **Gap:** `alpha_key_corners` (`alpha_key.py:175`) dispatches only `{'flood','threshold'}` (else-branch at `:205`, raise at `:206`); there is no ML path and no auto-recovery. The MCP `alpha_key` tool computes `keyed_ratio` (`mcp_server.py:241`) purely for the agent to read — nothing acts on it.
- **Surface:** Add `method="rembg"` to `alpha_key_corners`. New `cascade_img.curation.ml_key.remove_background_ml(img, model="u2net", post_process_mask=True) -> tuple[Image.Image, int]`. New MCP `alpha_key` kwargs: `auto_fallback: bool = False, fallback_low: float = 0.1, fallback_high: float = 0.9, fallback_model: str = "u2net"` — when the flood pass yields a ratio outside `[low, high]`, re-run via rembg and return whichever wins, tagged with which path produced it.
- **Implementation:** New `curation/ml_key.py`: `from rembg import remove, new_session` guarded by `try/except ImportError` raising a structured `MLExtraNotInstalled` (with `.remediation='pip install cascade-img[ml]'`, matching the pattern `_run_tool` reads at `mcp_server.py:99`). `remove()` takes/returns a PIL Image; convert to RGBA and count `alpha==0` via `getchannel('A').histogram()[0]` (same technique as `mcp_server.py:231`). In `alpha_key_corners`, add `elif method == "rembg": rgba, keyed = remove_background_ml(rgba, ...)` **before** the else at `:205`. The auto-fallback lives in the MCP tool's `go()` (`mcp_server.py:222`): run flood, then — because `alpha_key_corners` returns only `Image.Image`, not a count — **recompute `to_ratio` from the returned image** via `keyed.getchannel('A').histogram()[0]` exactly as the flood path does at `:231`, and if out-of-band and `auto_fallback`, call `alpha_key_corners(img, method='rembg')`; emit `ALPHA_KEY_FALLBACK` with `from_method`/`to_method`/`from_ratio`/`to_ratio`. The terminal `ALPHA_KEY_APPLIED` still fires inside `alpha_key_corners` for whichever method produced output, so the `keyed_count`/`total_count` contract holds.
- **Caveats:** `rembg` requires Python >=3.11,<3.14 and pulls `onnxruntime` + a downloaded model into `~/.u2net`; cascade-img sets `requires-python='>=3.10'`, so the `[ml]` extra is **not installable on 3.10** — document that the extra narrows the range, and the ImportError guard must also catch the version/`onnxruntime` mismatch. First call downloads the model (network + disk write); cache via `new_session()`. rembg has its own opinion about foreground and can crop a legitimately-background-colored subject region — keep flood the default, rembg opt-in/fallback only.
- **Deps:** new optional `[ml]` = `["rembg>=2.0.70", "onnxruntime>=1.17"]`. No core-dep change.
- **Test:** `tests/curation/test_ml_key.py`, `importorskip('rembg')`. Auto-fallback test: monkeypatch `remove_background_ml` to a stub, feed an image that floods to >0.9, assert `ALPHA_KEY_FALLBACK` then `ALPHA_KEY_APPLIED` with `to_method='rembg'`. Third test: ImportError guard raises `MLExtraNotInstalled` with `.remediation` when rembg is absent.
- **Source:** rembg README — Python >=3.11,<3.14, `onnxruntime` core dep, `remove()` takes/returns PIL Image, models u2net/isnet/birefnet.

#### Automated candidate scoring (`score_grid`)

- **Gap:** No scoring anywhere in `curation/`. `crop_quadrant` returns one quadrant with no quality assessment; the agent compares visually. The MCP layer exposes crop/key/promote but nothing that evaluates candidates (`mcp_server.py:183`).
- **Surface:** New `cascade_img.curation.score.score_grid(src, key_for_transparency=False, tolerance=DEFAULT_TOLERANCE) -> list[QuadrantScore]`, where `QuadrantScore` is `dataclass(quadrant, sharpness, contrast, edge_density, transparency_cleanliness, composite)`, sorted by `composite` desc. MCP `score_grid(src, key_for_transparency=False) -> {ok, result:{ranked:[...], best_quadrant}}`.
- **Implementation:** New `curation/score.py`. Reuse `crop_grid.crop_quadrant(src, q)` for q in 1..4 — **do not** reimplement the split, so `QUADRANT_OFFSETS` (`crop_grid.py:22`) stays single-sourced. Pure-PIL metrics: sharpness = variance of `ImageFilter.FIND_EDGES`; contrast = `ImageStat.Stat(gray).stddev[0]`; edge_density = fraction of `FIND_EDGES` pixels above a threshold via `point()`+histogram; `transparency_cleanliness` (only when `key_for_transparency`) = alpha-channel bimodality from `getchannel('A').histogram()` after running `alpha_key_corners(quad, tolerance)`. `composite` = documented weighted sum. Emit `GRID_SCORED` once with the per-quadrant arrays + `best_quadrant`. Slots **before** crop: score → read `best_quadrant` → `crop_grid(quadrant=best)`.
- **Caveats:** `numpy`/`cv2` would give a true `cv2.Laplacian` variance and Sobel edges in C but add a heavy non-core dep for marginal gain on 1024px quadrants — keep pure-PIL default; offer a `cv2` path under `[ml]` only if profiling shows a bottleneck. Scores are relative within one grid, not absolute; the composite weights are a documented judgment call, not gospel. `transparency_cleanliness` double-keys (scorer keys, then the real tool keys again) — acceptable since scoring is read-only, but note the duplicated work.
- **Deps:** none for the pure-PIL default; optional `cv2` under `[ml]`.
- **Test:** `tests/curation/test_score.py` — synthetic grid with one sharp high-contrast checkerboard quadrant and three flat gray; assert `score_grid` returns the checkerboard as `best_quadrant` and its sharpness/edge_density/contrast strictly exceed the flats'; `snapshot()[-1]['tag']=='GRID_SCORED'` with a 4-element array. Second test: `key_for_transparency=True` on a clean-sprite-vs-noisy-bg grid ranks the clean quadrant higher.
- **Source:** Pillow `ImageFilter.FIND_EDGES`, `ImageStat.Stat`, `Image.histogram`/`getchannel` (stable public API); MJ quadrant semantics confirmed.

#### Auto-trim / content bounding-box crop

- **Gap:** Nothing trims. `alpha_key_corners` (`alpha_key.py:175`) returns a full-frame RGBA with the subject floating in transparency; `promote` (`promote.py:15`) byte-copies the full frame.
- **Surface:** New `cascade_img.curation.trim.trim_to_content(img, padding=0, alpha_threshold=0) -> Image.Image`. MCP `auto_trim(src, dest, padding=0) -> {ok, result:{dest, w, h, orig_w, orig_h, trimmed_px}}`.
- **Implementation:** New `curation/trim.py`. Default path = `img.convert('RGBA').getbbox()` — `getbbox` already keys off non-zero regions, and `getbbox(alpha_only=True)` returns the content box directly, so the manual `getchannel('A')` + `point()` binarize is **redundant** for the default. The threshold path (optional halo suppression) = `img.getchannel('A').point(lambda a: 255 if a > threshold else 0).getbbox()`. If `getbbox` is None (fully transparent), return unchanged and emit `trimmed_px=0`. Expand by `padding` (clamped to bounds), then `img.crop(box)`. Emit `CONTENT_TRIMMED` with orig/new dims and `trimmed_px`. Register `CONTENT_TRIMMED` in `versions/0.1.json` under CurationKit (`:64`) or the parity gate fails. Slots after key, before promote.
- **Caveats:** `getbbox` on a noisy key (alpha 1–30 halo) under-trims unless `alpha_threshold` is set; default 0 trims only fully-keyed margin (safe conservative). Trimming destroys the original frame's anchor/registration — make it opt-in, and note that atlas packing should run on trimmed sprites but then record the original offsets in the frame map.
- **Deps:** none — `getchannel`/`getbbox`/`point`/`crop` are core.

#### Palette quantization

- **Gap:** No quantization step. `alpha_key` produces full-color RGBA; `promote` byte-copies it. Nothing between key and promote constrains the palette, even though the README sells a pixel-art use case.
- **Surface:** New `cascade_img.curation.palette.quantize_palette(img, colors=16, method="maxcoverage", dither=False, preserve_alpha=True) -> Image.Image`. MCP `quantize(src, dest, colors=16, method="maxcoverage", dither=False) -> {ok, result:{dest, colors_requested, colors_actual, method, w, h}}`.
- **Implementation:** New `curation/palette.py`. Map method → `Image.Quantize` enum: maxcoverage→MAXCOVERAGE, median→MEDIANCUT, octree→FASTOCTREE, libimagequant→LIBIMAGEQUANT. **MEDIANCUT/MAXCOVERAGE do not support RGBA** (in the official docs; FASTOCTREE is the RGBA default) — so when `preserve_alpha` and input is RGBA, split off alpha, quantize the RGB, then re-attach alpha. `dither` maps to `Image.Dither.FLOYDSTEINBERG` vs `NONE` (NONE is the pixel-art default). `colors_actual = len(result.getcolors(maxcolors=256) or [])` — the quantized image is already mode 'P', so the `.convert('P')` in the original sketch is a redundant no-op. Gate the LIBIMAGEQUANT branch on `PIL.features.check_feature('libimagequant')` (real API) before reaching the enum, since standard wheels ship without it. Emit `PALETTE_QUANTIZED`; add it to `versions/0.1.json` CurationKit (`:64`). Slots after trim, before promote/convert.
- **Caveats:** The RGBA/MEDIANCUT incompatibility is the trap — the split-quantize-reattach path is mandatory under `preserve_alpha`. Quantize **after** keying (quantizing before would fold the background into the palette and break `_sample_bg`'s corner-average, `alpha_key.py:57`). MAXCOVERAGE as the RGB pixel-art default is **community consensus, not vendor documentation** — do not cite the docs for that claim.
- **Deps:** none for MEDIANCUT/MAXCOVERAGE/FASTOCTREE; LIBIMAGEQUANT only on a Pillow built with it (detect and error cleanly).
- **Test:** `tests/curation/test_palette.py` — smooth 64×64 RGB gradient, `quantize_palette(colors=8, method='maxcoverage')`; assert `result.convert('RGB').getcolors()` ≤ 8 and `PALETTE_QUANTIZED` with `colors_requested==8`, `colors_actual<=8`. Second: a keyed RGBA sprite with `preserve_alpha=True` still has corner alpha==0 after quantize.

#### Near-duplicate detection (perceptual hashing)

- **Gap:** No similarity awareness. `promote` (`promote.py:15`) duplicates or overwrites with no signal that two candidates or a candidate-vs-promoted-asset are perceptually the same. `keyed_ratio` is the only quality signal and says nothing about duplication.
- **Surface:** New `cascade_img.curation.dedup.phash(img, hash_size=8) -> int` and `is_near_duplicate(a, b, max_distance=5) -> tuple[bool, int]`. MCP `check_duplicate(src, against: list[str], max_distance=5) -> {ok, result:{src_hash, matches:[{path, distance, is_duplicate}], any_duplicate}}`.
- **Implementation:** New `curation/dedup.py`, pure-PIL dHash: `convert('L').resize((hash_size+1, hash_size), LANCZOS)`, compare adjacent pixels → bitstring → int. Hamming distance = `bin(a^b).count('1')`. `check_duplicate` hashes `src` once and each path in `against`, returns sorted matches. Emit `DUPLICATE_CHECKED` — this tag is **not** in `versions/0.1.json` (grep count 0), so emitting it as-is raises at the validator (`_runtime.py:65`); add it under a curation stratum/category alongside CROP_QUADRANT/ALPHA_KEY_APPLIED/ASSET_PROMOTED, declare its payload (`candidate_hash`/`compared_count`/`min_distance`/`any_duplicate`), and keep the tag/callsite ledger consistent. - **Caveats:** Perceptual hashing is intentionally lossy — `max_distance` trades false-positive against false-negative; 5/64 bits is a reasonable, documented, tunable default. Default to **dHash** over aHash (aHash is brightness-fragile). - **Deps:** none — pure-PIL `convert`/`resize` + stdlib bit ops; optional `imagehash` under `[ml]`.
- **Test:** `tests/curation/test_dedup.py` — hash an image and a 1-pixel-perturbed copy, assert `is_near_duplicate` → `(True, small)`; two distinct images (checkerboard vs solid) → `(False, large)`; for the MCP tool, write two near-identical PNGs + one distinct, call `check_duplicate(src=near1, against=[near2, distinct])`, assert `near2` flagged `is_duplicate=True`, `distinct` False, `any_duplicate=True`, and `DUPLICATE_CHECKED` with `compared_count==2`.

### Backend & operability

The bridge is a single-process Flask + discord.py-self daemon whose entire job state lives in an in-RAM `OrderedDict` (`JOBS`, `bridge.py:317`, guarded by `LOCK`/`TERMINAL_CV` at `:319-320`, evicted by `_evict_if_needed` at `:360`). Clean and well-instrumented, but with production-grade gaps: a restart silently drops every in-flight job; non-MCP clients can only long-poll; nothing rate-limits submissions; `/health` reports only five fields; and `Config` (`:104`) hard-codes one token/channel. Each feature below reuses the existing `LOCK`/`TERMINAL_CV`/eviction/emit machinery by name and respects the locked-vocabulary parity discipline.

| Feature | What it buys | Proposed surface | Effort | Deps |
|---|---|---|---|---|
| Persistent job store (SQLite) | Survive restart; stop dropping in-flight jobs | `JobStore(path)` + write-through `touch()` | M | none (stdlib) |
| SSE completion push (`/events`) | Low-latency push, no long-poll herd | `GET /events?job_id=&since=` | M | none |
| Seed-locked reroll registry | Controlled iteration; re-roll lineage | `--seed` + `reroll(parent_job_id, seed_lock, **overrides)` | S | none |

| Multi-account rotation | Failover on ban; spread load | `AccountProfile` pool + `rotate_account(reason)` | L | none (Phase 1–2) |

#### Multi-account / multi-channel config with sacrificial-account rotation

- **Gap:** `Config` (`bridge.py:104-116`) holds exactly one `discord_token`/`channel_id`/`guild_id`/`mj_imagine_version`/`mj_imagine_command_id`. The `discord.Client`, `_ready` Event, reconnect loop, and every interaction (`_send_imagine` `:842`, `_press_button` `:880`) read a single `c = _cfg()`. On terminal auth failure the reconnect loop emits `DISCORD_RECONNECT_FAILED(auth)` and **exits**, leaving the daemon permanently dead-to-Discord with no failover (`_is_terminal_auth_failure` at `:1127`).
- **Surface:** New `@dataclass AccountProfile {name, discord_token, channel_id, guild_id, mj_imagine_version, mj_imagine_command_id}` and `Config.accounts: list[AccountProfile]` loaded from the flat env (single-account back-compat) or a `CASCADE_ACCOUNTS_FILE` JSON/TOML. New `class AccountPool`: `.active()`, `.rotate(reason)`, `.assign_for_job()` (round-robin). Each `Job` gains `account_name`. `/health` reports `accounts: [{name, discord_ready, inflight, state}]`. New MCP tool `rotate_account(reason) -> {from, to}`.
- **Implementation (phased):** **Phase 1 (config, S):** `AccountProfile` + `Config.accounts` with the single-env path producing a one-element list; thread `Job.account_name` (default `accounts[0].name`); `/health` lists the one account. No behavior change. **Phase 2 (failover, M):** in `_is_terminal_auth_failure` handling, instead of exiting, call `AccountPool.rotate('auth')` and continue the reconnect loop with the next profile's token; emit `ACCOUNT_ROTATED {from_account, to_account, reason}`. **Phase 3 (multi-client, L):** run one `discord.Client` + `_ready` Event + asyncio loop per account (the current module-global `client`/`_ready`/`_loop_holder` become per-profile, keyed by `account_name`); `_send_imagine`/`_press_button` select by `job.account_name`; the `on_message` channel check (`_ingest_message`, `:552`, currently `message.channel.id != c.channel_id`) must check against the right profile's channel. `_match_grid`/`_match_upscale` (`:477`/`:504`) already key on `request_token`, which is account-independent, so cross-account mis-routing is avoided **as long as** each account watches a distinct channel.
- **Caveats:** Phase 3 multiplies WebSocket/loop state and is genuinely large — the entire Discord side currently assumes module-global singletons. `_match_grid` routing keys on the account-independent token, so the only cross-account hazard is the channel check at `:552`. The semaphore from the rate-limiter feature must become **per-account** (budgets are per-account). Do not auto-rotate on a single 429 (transient); rotate only on terminal auth failure or sustained 429s. Ship Phase 1+2 first; Phase 3 only if real multi-account throughput is needed.
- **Deps:** none for Phase 1–2 (`json`/`tomllib`, stdlib 3.11+). Phase 3 adds discord.py-self client instances, no new packages.
- **Test:** Phase 1 — load a two-profile `CASCADE_ACCOUNTS_FILE`, assert `Config.accounts` has both; assert the single-env path yields a one-element list (extend `tests/backends/midjourney_discord/test_config.py`, which uses the `valid_env` fixture). Phase 2 — monkeypatch `_is_terminal_auth_failure` True for account[0], drive `_run_discord` one iteration, assert the pool advanced to account[1] and `ACCOUNT_ROTATED(reason='auth')` fired. Phase 3 — two accounts, two jobs round-robin, assert each `Job.account_name` set and `_match_grid` still routes by token. `rotate_account` MCP test mirrors `test_mcp_server.py` envelope assertions.

## 9. Code & code-vs-docs notes

The engine is carefully written: small surfaces, honest sync/async ABC, structured error envelopes, thread-safe `PromptLog`, correct FD handling in `crop_quadrant` and the `alpha_key` tool, and a solid resilience layer (backoff reconnect, per-slot upscale isolation via `gather(return_exceptions=True)`, token-based grid routing, `Condition`-based `/wait`, LRU+TTL eviction). `ruff check src/` is clean and `py.typed` ships. The flood-fill keyer matches its docstring and the `alpha_key` `keyed_ratio` failure contract is accurate. Two findings are serious:

1. **`cascade-mj` awaits synchronous backend methods — the live path raises `TypeError` [confirmed].** `backends/base.py:30-31` states methods are synchronous at v0.1, and `MidjourneyDiscordBackend.imagine/.wait` are plain `def`s returning `dict`. Fix: `await asyncio.to_thread(backend.imagine, …)` / `…(backend.wait, …)`; add a `dry_run=False` test with a stubbed sync backend.

2. **The committed test config fails to collect on the supported toolchain [confirmed].** `pyproject.toml:114-121` sets `filterwarnings = ["error", …]` with module-scoped ignores for `discord`/`mcp`/`pydantic` only. `discord.py-self` imports `audioop`, whose `DeprecationWarning` is attributed to module `discord.player`, so the ignore misses it and `pytest -q` errors with "11 errors during collection" on 3.11; only `-W ignore::DeprecationWarning` reaches "113 passed." **Correction to the proposed fix:** the reviewer's suggested `ignore::DeprecationWarning:audioop` does *not* work; the filters that actually fix it are `ignore::DeprecationWarning:discord.player` or message-anchored `ignore:'audioop' is deprecated:DeprecationWarning`. Drop the 3.13 classifier or pin a workaround, and verify green on 3.11/3.12 before re-asserting "113/113 green."
