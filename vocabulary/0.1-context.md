# vocabulary/0.1-context.md — cascade-img vocabulary v0.1

*A plain-language companion to `vocabulary/0.1.json`: what each tag and layer means, kept in sync with the file. The JSON is the source of truth; this doc explains it. v0.1 locked 2026-06-02.*

---

## Intent and scope

This vocabulary names the events cascade-img emits at runtime — what the bridge daemon does, what the MCP server does, what the CLI does, what the curation primitives do.

We have found that doing so makes LLM-assisted development easier, because the code is designed around a fixed event grammar rather than ad-hoc logging. Because every runtime action maps to a named event with a typed payload, a reader or an agent can reconstruct what the program did from the event stream alone, instead of parsing free-text prose logs.

In scope: the v0.1 surface (MJ Discord backend + composer + curation + log + MCP + CLI). Out of scope: future backends (Flux, DALL-E, Imagen) — those bring their own backend-specific tags at v0.3+.

---

## Per-layer decisions

### Layer 0 — Ontology (entities)

The durable things cascade-img tracks state about. Naming them is what tells a reader what every `asset_id` or `job_id` in a tag's payload refers to.

- **Job** — one /imagine roll through to its terminal state (done | failed). Identified by `job_id`. Most tags carry both `job_id` and `asset_id` pointing back to this entity.
- **Asset** — a named output the user is generating (e.g. `"mountain-icon"`, `"hero-portrait"`). Identified by `asset_id`. Carried in every job-category tag's payload.
- **Bridge** — the long-running Flask + discord.py-self daemon process. Operated through HTTP. Most config-category and bridge-category tags concern it.
- **Backend** — the pluggable backend implementing `ImageGenerationBackend`. v0.1 has one (`MidjourneyDiscordBackend`). The `backend` payload field on `CASCADE_INIT` carries which is active.

These four are listed explicitly in `0.1.json` under `ontology.entities`. (A second backend may later force cross-backend entity reconciliation — a v0.2 concern.)

### Layer 1 — Lexical (tag names)

Fixes the spelling and naming form of every tag, so the same event always carries the same identifier.

47 tags across 11 categories (session, config, bridge, job, discord, backend, curation, composer, log, mcp, cli). Names follow the `{ENTITY|SUBSYSTEM}_{VERB_PAST_PARTICIPLE}` form. `tags[]` in the JSON is the full list; a representative sample:

- `CASCADE_INIT` (session start)
- `CONFIG_VALIDATED` / `CONFIG_VALIDATION_FAILED`
- `DISCORD_CONNECTED` / `DISCORD_DISCONNECTED` / `DISCORD_RECONNECTING` / `DISCORD_RECONNECT_FAILED`
- `BRIDGE_HEALTHY` / `BRIDGE_CHECKENV_RAN` / `BRIDGE_DOCTOR_RAN` / `BRIDGE_SHUTDOWN`
- `IMAGINE_FIRED`, `GRID_MATCHED`, `GRID_RECEIVED`, `UPSCALE_REQUESTED`, `UPSCALE_RECEIVED`, `JOB_COMPLETED`, `JOB_FAILED`, `JOB_EVICTED`
- `MJ_ACTION_REQUESTED` / `MJ_ACTION_FAILED` / `MJ_DERIVED_RECEIVED` / `MJ_DERIVED_FAILED`
- `BACKEND_HTTP_CALLED`
- `CROP_QUADRANT`, `ALPHA_KEY_APPLIED`, `ASSET_PROMOTED`, `CONTACT_SHEET_RENDERED`, `AUTO_TRIM_APPLIED`, `PALETTE_QUANTIZED`, `SPRITE_SHEET_PACKED`, `GRID_SCORED`
- `PROMPT_COMPOSED`, `PROMPT_LOGGED`
- `MCP_SERVER_STARTED` / `MCP_SERVER_STOPPED`, `MCP_TOOL_CALLED`, `MCP_TOOL_COMPLETED`, `MCP_TOOL_FAILED`
- `CLI_ROLL_STARTED`, `CLI_ROLL_COMPLETED`, `CLI_ROLL_FAILED`

The imagine-to-promote loop fires as `IMAGINE_FIRED → GRID_MATCHED → GRID_RECEIVED → (UPSCALE_REQUESTED → UPSCALE_RECEIVED)* → JOB_COMPLETED`.

Past-tense forms enforced: `RECEIVED` not `RECEIVE`; `MATCHED` not `MATCH`; `VALIDATED` not `VALIDATE`. Irregular forms handled explicitly (`COMPLETED` not `COMPLETE` to match past-tense convention).

### Layer 2 — Payload (typed fields per tag)

Declares the typed fields each tag carries, so every event has a known shape rather than an open bag of data.

Required fields named per tag; `optional_payload` is an empty array on every v0.1 tag (none carry optional fields yet). The `asset_id` and `job_id` fields point back to the Layer 0 entities above.

The `match_path` field on `GRID_MATCHED` carries `"pending"` (first-touch match), `"progress_fallback"` (the V7 patch path), or `"unknown"`, recording which match path fired.

The `has_*` fields (e.g. `has_guild_id`, `has_job_id`, `has_error`) carry boolean shape information without leaking the underlying values. A missing guild_id surfaces via `CONFIG_VALIDATED.has_guild_id=false` rather than as a free-text warning.

### Layer 3 — Session (event stream framing)

Marks where an event stream begins and ends, so individual events can be grouped into one process or tool session. Declared in `sessions.strata`:

- **`bridge_session`** — opens on `CASCADE_INIT`, closes on `BRIDGE_SHUTDOWN`; persistent attributes `package_version` and `backend`. Frames the whole daemon lifecycle: config, the Discord connection, every job, eviction, and derived results. (`CONFIG_VALIDATION_FAILED` can still end the process before the session formally begins.)
- **`mcp_session`** — opens on `MCP_SERVER_STARTED`, closes on `MCP_SERVER_STOPPED`; persistent attribute `transport` (`stdio` | `http`). Frames the MCP tool calls.

### Layer 4 — Temporal (timing invariants)

Records the expected timing between events, so a stalled or slow run can be distinguished from a normal one. Declared in `temporal_invariants.invariants`:

- **Window** — `IMAGINE_FIRED → JOB_COMPLETED` within 180s (grid-only), 360s (single upscale), or 600s (`upscale=all`): the three upscale-mode bands.
- **Pairing** — every `MCP_TOOL_CALLED` resolves to one `MCP_TOOL_COMPLETED` or `MCP_TOOL_FAILED`.
- **Forbidden-after** — once `JOB_COMPLETED` or `JOB_FAILED` fires, no further `GRID_*` / `UPSCALE_*` tag carries the same `job_id`.

### Layer 5 — State-Transition (ordering graph)

Defines which events must precede or follow which others, so out-of-order sequences can be flagged as invalid. Declared in `state_transitions.rules`:

- **Forced-next** — `CASCADE_INIT → CONFIG_VALIDATED` (or the process exits via `CONFIG_VALIDATION_FAILED`).
- **Pairing-ordering** — `DISCORD_CONNECTED` before any `IMAGINE_FIRED` (gated by `_ready.is_set()`); `GRID_MATCHED → GRID_RECEIVED` for the same `job_id`; `UPSCALE_REQUESTED → UPSCALE_RECEIVED` for the same `job_id`+`slot`.
- **Terminal** — `JOB_COMPLETED` and `JOB_FAILED` end a `job_id`'s timeline; no later job-category tag carries it.

### Layer 6 — Runtime / Operator

Identifies which component emits which tags, so every event traces back to the part of the system that produced it. Declared in `operators.operators`, each with its emit set and source location:

- **`MidjourneyDiscordBackend`** (`backends/midjourney_discord/bridge_client.py`) — `BACKEND_HTTP_CALLED`.
- **`BridgeDaemon`** (`backends/midjourney_discord/bridge.py`) — the session/config/discord/bridge/job tags: `CASCADE_INIT`, `CONFIG_*`, `DISCORD_*`, `BRIDGE_*`, `IMAGINE_FIRED`, `GRID_*`, `UPSCALE_*`, `JOB_*`, `MJ_ACTION_*`, `MJ_DERIVED_*`.
- **`PromptComposer`** (`prompt/composer.py`) — `PROMPT_COMPOSED`.
- **`PromptLog`** (`prompt/prompt_log.py`) — `PROMPT_LOGGED`.
- **`CurationKit`** (`curation/`) — `CROP_QUADRANT`, `ALPHA_KEY_APPLIED`, `ASSET_PROMOTED`, `CONTACT_SHEET_RENDERED`, `AUTO_TRIM_APPLIED`, `PALETTE_QUANTIZED`, `SPRITE_SHEET_PACKED`, `GRID_SCORED`.
- **`MCPServer`** (`interfaces/mcp/`) — `MCP_SERVER_STARTED`, `MCP_SERVER_STOPPED`, `MCP_TOOL_*`.
- **`AssetCLI`** (`interfaces/cli/generate_image.py`) — `CLI_ROLL_*`.

### Layer 7 — Evidence (payload constraints beyond type signatures)

Pins specific payload fields to a fixed set of allowed values, so any value outside that set is rejected. Declared in `evidence_constraints.constraints` and **enforced at emit**: `Vocabulary.validate()` raises on a value outside a field's locked enum, not just on an unknown tag or a missing field. The constrained fields:

- `JOB_FAILED.error_code` — the stable failure-code set (`DISCORD_400_OUTDATED`, `DISCORD_400_UNKNOWN_CHANNEL`, `DISCORD_401`, `DISCORD_HTTP_ERROR`, `DISCORD_NOT_READY`, `GRID_DOWNLOAD_FAILED`, `MJ_UUID_MISSING`, `SUBMIT_FAILED`, `UPSCALE_DOWNLOAD_FAILED`, `UPSCALE_BUTTON_FAILED`, `UPSCALE_ALL_BUTTONS_FAILED`, `RESUBMIT_REQUIRED`).
- `CONFIG_VALIDATION_FAILED.code` — `MISSING_DISCORD_TOKEN`, `MISSING_CHANNEL_ID`, `INVALID_CHANNEL_ID`, `MISSING_IMAGINE_VERSION`, `INVALID_PORT`.
- `MCP_TOOL_*.tool` — the 16 MCP tool names.
- `GRID_MATCHED.match_path` — `pending` | `progress_fallback` | `unknown`.
- `MCP_SERVER_STARTED.transport` — `stdio` | `http`.
- `MJ_ACTION_*.action` — the response-message action set (`upscale_*`, `vary_*`, `zoom_out_*`, `pan_*`, `animate_*`, `favorite`).
- `MJ_DERIVED_*.action_kind` — `variation` | `zoom` | `pan` | `upscale` | `animation`.

Each constraint is declared `enforceability: mechanical`; `0.1.json` is the source of truth for the exact value sets.

### Layers 8-10 — Report / Version / Grammar-Growth

Cover how events roll up into reports, how the vocabulary is versioned and locked, and how it is allowed to grow over time.

- **Layer 8 (Report binding):** `report_binding` lists every category as an `observed_category` and `incident` as the `notes_category`. No view tags (cascade-img has no UI).
- **Layer 9 (Version):** `0.1`, `locked: true`, locked 2026-06-02 by Green Rose Systems, `prior_version: null`.
- **Layer 10 (Grammar-Growth):** `grammar_growth.project_overrides` declares `validator-extras: strict` (undeclared payload fields and out-of-enum values raise at emit; production may relax the extras check via `CASCADE_STRICT_SIGNALS=false`) and `view-payload-universal: n/a` (no view category).

## Tag-to-artifact table

cascade-img has no UI; its observable outputs are FILE OUTPUTS (PNGs written to disk) and HTTP RESPONSES (structured JSON returned to clients). This table pairs each behavior tag with the artifact it corresponds to:

| Behavior tag | Artifact counterpart | Notes |
|---|---|---|
| `IMAGINE_FIRED` | HTTP response: `{job_id, asset_id, status: "submitted", upscale}` from `POST /imagine` | Tag fires after the response is returned. |
| `GRID_MATCHED` | None at the consumer surface — internal state transition | Visible only in the event stream. |
| `GRID_RECEIVED` | PNG file at `{output_dir}/{asset_id}.{png|webp}` | The `path` and `bytes` payload fields identify the artifact. |
| `UPSCALE_RECEIVED` | PNG file at `{output_dir}/{asset_id}_u{slot}.{png|webp}` or `.png` | Same shape. |
| `JOB_COMPLETED` | HTTP response: `{status: "done", image_path, grid_path, upscale_paths}` from `GET /wait` | Tag fires before the response is returned. |
| `JOB_FAILED` | HTTP response: `{status: "failed", error, error_code}` from `GET /wait` | Stable `error_code` mapped to remediation. |
| `MJ_DERIVED_RECEIVED` | PNG/WebP at `{output_dir}/{asset_id}_{kind}_{uuid8}.{ext}` | A vary/zoom/pan/upscale/animation result from an `mj_action` press. |
| `ASSET_PROMOTED` | PNG file at consumer's `dest` path | Idempotent — overwrites. |
| `PROMPT_LOGGED` | JSONL line in `{prompt_log_path}` | Append-only. |

## Project-specific notes

- **Strict validation:** at emit, the vocabulary checks the tag exists, every required field is present, no undeclared field is passed, and every enum-constrained field carries an allowed value — each violation raises. `CASCADE_STRICT_SIGNALS=false` relaxes this in production.
- **Vocabulary location:** canonical at project root `vocabulary/0.1.json` AND bundled in the package at `packages/python/src/cascade_img/vocabulary/versions/0.1.json` (the package-data copy for the wheel). The two are kept identical; the CI parity step reads the package-data copy.
