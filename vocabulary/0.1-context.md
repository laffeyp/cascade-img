# cascade-img event vocabulary v0.1 — how to read the event stream

*A companion to [`0.1.json`](0.1.json), the catalog this explains. [`0.1-reference.md`](0.1-reference.md) is the generated per-tag listing — every tag's payload fields, emitter, and allowed values; this document explains how the events fit together and how the catalog is laid out.*

## What this is

cascade-img names every event it emits at runtime. The catalog is a fixed set of 48 events across 11 categories — the bridge daemon, the MCP server, the CLI, the prompt composer, and the curation primitives — and each event carries a typed payload (`IMAGINE_FIRED` carries `asset_id`, `job_id`, `prompt_chars`, `upscale`, and nothing else).

Two properties make the event stream a reliable account of what the program did, rather than a best-effort log:

- **Validation happens at the speaker.** `emit()` checks every event against the catalog and raises on an unknown tag, a missing required field, an undeclared field, or a value outside a locked enum. A malformed event never reaches the stream — it fails the call that tried to emit it. (`CASCADE_STRICT_SIGNALS=false` downgrades this to log-and-continue in production.)
- **The legal order of events is declared, and checked.** The catalog states which events must precede which — per job, not globally — and how long a run should take. A recorded trace replays against those rules mechanically, with no human reading required.

So a person or an agent reconstructs a run from the stream alone — which job fired, whether its grid arrived before its upscales, whether it finished inside its timing band — by reading known tokens with typed fields instead of parsing prose.

In scope is the v0.1 surface: the Midjourney-via-Discord backend, the composer, the curation primitives, the prompt log, the MCP server, and the CLI. A future backend brings its own events — for example `FLUX_FIRED` / `FLUX_RECEIVED` / `FLUX_FAILED` when a Flux backend lands — added the same way (see *Growing the catalog*).

## Entities

Every `job_id` or `asset_id` in a payload points at one of four durable things (declared under `ontology.entities` in the catalog):

- **Job** — one `/imagine` roll through to its terminal state (done or failed). Identified by `job_id`.
- **Asset** — a named output the caller is generating (e.g. `"mountain-icon"`, `"hero-portrait"`). Identified by `asset_id`.
- **Bridge** — the long-running Flask + discord.py-self daemon process, operated over HTTP.
- **Backend** — the component implementing `ImageGenerationBackend`. v0.1 has one, `MidjourneyDiscordBackend`; the `backend` field on `CASCADE_INIT` records which is active.

## The lifecycle

A normal generation run produces this sequence:

```
CONFIG_VALIDATED → CASCADE_INIT → DISCORD_CONNECTED
  → IMAGINE_FIRED → GRID_MATCHED → GRID_RECEIVED
  → (UPSCALE_REQUESTED → UPSCALE_RECEIVED)*   # zero, one, or four pairs
  → JOB_COMPLETED
```

The daemon validates its config (`CONFIG_VALIDATED`) before it announces startup (`CASCADE_INIT`); `--check-env` and `--doctor` validate and exit, so `CONFIG_VALIDATED` can appear without a following `CASCADE_INIT`. Daemon shutdown ends the stream with `BRIDGE_SHUTDOWN`. An MCP session is framed the same way: `MCP_SERVER_STARTED` opens it, each tool call fires `MCP_TOOL_CALLED` then exactly one of `MCP_TOOL_COMPLETED` / `MCP_TOOL_FAILED`, and `MCP_SERVER_STOPPED` closes it. A CLI roll wraps the job sequence in `CLI_ROLL_STARTED` / `CLI_ROLL_COMPLETED` (or `CLI_ROLL_FAILED`).

Order is tracked **per `job_id`**, so two concurrent jobs never interfere: `GRID_RECEIVED` must follow `GRID_MATCHED` for the *same* job, and `UPSCALE_RECEIVED` must follow `UPSCALE_REQUESTED` for the same job *and* slot.

Each run also has a timing band for `IMAGINE_FIRED → JOB_COMPLETED`, selected by the run's `upscale` mode: 180s when `upscale` is none (grid only), 360s for a single `1`–`4` upscale, 600s for `all`. A run past its band is flagged as a **warning**, not a failure — a slow Midjourney day is worth noticing, not worth failing a check over.

## How the catalog is laid out

`0.1.json` is more than a tag list. Its top-level keys, each a section of the file:

- **`tags`** — the 48 events. Each entry is the tag name, its category, its required `payload` fields, any `optional_payload`, and a `note`.
- **`ontology`** — the four entities above.
- **`sessions`** — the two framing pairs: `CASCADE_INIT … BRIDGE_SHUTDOWN` (the daemon, carrying `package_version` + `backend`) and `MCP_SERVER_STARTED … MCP_SERVER_STOPPED` (carrying `transport`: `stdio` | `http`).
- **`state_transitions`** — which events must precede which, and the slice key (`job_id`, `job_id`+`slot`, or none) each rule applies per. Includes the terminal rule: after `JOB_COMPLETED` / `JOB_FAILED`, no further lifecycle event carries that `job_id`.
- **`temporal_invariants`** — the timing bands above, each tagged with the `upscale` mode it applies to and a `severity`.
- **`evidence_constraints`** — the fields whose values are locked to an enum (e.g. `JOB_FAILED.error_code`, `GRID_MATCHED.match_path`), enforced at emit.
- **`operators`** — a map from every tag to the component and source file that emits it, so any event traces back to code.
- **`report_binding`** and **`grammar_growth`** — bookkeeping: which categories a summary groups by, and the policy for adding events (see *Growing the catalog*).

## How it's enforced

Three checks, at build time and run time, hold the stream to the catalog.

**Emit-time validation.** `Vocabulary.validate()` runs inside `emit()` (above), so the contract is enforced at the one place every event passes through.

**The trace checker.** `cascade-trace-check` reads a recorded trace — one JSON event per line, written whenever `CASCADE_EVENT_LOG` is set — and runs the catalog's sequence and timing rules against it. The rules are data, not code. This is the one that says a grid must be matched before it is received:

```json
{"kind": "pairing_ordering", "from_tag": "GRID_MATCHED",
 "to_tags_allowed": ["GRID_RECEIVED"], "payload_match": "job_id"}
```

The checker groups the trace's events by the rule's `payload_match` key — here `job_id` — and within each job reports a `GRID_RECEIVED` with no preceding `GRID_MATCHED` as an error. Because the grouping is per job, one job's events can never satisfy or violate another's. The terminal rule reads the same way over its own fields: it lists the lifecycle tags that may not reappear for a job once it has hit `JOB_COMPLETED` / `JOB_FAILED` (`forbidden_tags_after`) and the few that legitimately may (`exempt_tags` — an `mj_action` press and its derived result, an eviction, a late path collision). A timing-band overrun is reported at `severity: "warning"`; everything else is an error. The test suite runs the checker on every change with no credentials, and against each live end-to-end run.

**Parity and the catalog hash.** `check_vocabulary_parity.py` walks the source tree and fails if any `emit()` names a tag the catalog doesn't declare — an event must exist in the catalog before code may emit it. The same check recomputes a `sha256` over the catalog's tag names and required payloads (`tag_set_hash`) and fails if it differs from the stored value, so a tag or payload can't change without the change being acknowledged in the same commit.

## When things go wrong

Events marked `incident` report failures. Two carry stable, enum-locked codes:

- **`JOB_FAILED.error_code`** — `DISCORD_400_OUTDATED`, `DISCORD_400_UNKNOWN_CHANNEL`, `DISCORD_401`, `DISCORD_HTTP_ERROR`, `DISCORD_NOT_READY`, `GRID_DOWNLOAD_FAILED`, `MJ_UUID_MISSING`, `SUBMIT_FAILED`, `UPSCALE_DOWNLOAD_FAILED`, `UPSCALE_BUTTON_FAILED`, `UPSCALE_ALL_BUTTONS_FAILED`, `RESUBMIT_REQUIRED`. The same code appears in the `GET /wait` HTTP response, so a client maps it straight to remediation.
- **`CONFIG_VALIDATION_FAILED.code`** — `MISSING_DISCORD_TOKEN`, `MISSING_CHANNEL_ID`, `INVALID_CHANNEL_ID`, `MISSING_IMAGINE_VERSION`, `INVALID_PORT`. This one can end the process before `CONFIG_VALIDATED` ever fires.

Partial-failure incidents — `UPSCALE_PRESS_FAILED`, `UPSCALE_DOWNLOAD_DROPPED`, `JOB_SUBMIT_TIMEOUT`, `MJ_DERIVED_FAILED`, `OUTPUT_PATH_COLLISION` — do **not** terminate the job; each tag's note in the reference says what survives and what the client should do.

## Events and their artifacts

cascade-img has no UI; its observable outputs are files (PNGs on disk) and HTTP responses (structured JSON). This table pairs each behavior event with the artifact it corresponds to:

| Behavior tag | Artifact counterpart | Notes |
|---|---|---|
| `IMAGINE_FIRED` | HTTP response: `{job_id, asset_id, status: "submitted", upscale}` from `POST /imagine` | Tag fires after the response is returned. |
| `GRID_MATCHED` | None at the consumer surface — internal state transition | Visible only in the event stream. |
| `GRID_RECEIVED` | PNG file at `{output_dir}/{asset_id}.{png|webp}` | The `path` and `bytes` payload fields identify the artifact. |
| `UPSCALE_RECEIVED` | PNG file at `{output_dir}/{asset_id}_u{slot}.{png|webp}` or `.png` | Same shape. |
| `JOB_COMPLETED` | HTTP response: `{status: "done", image_path, grid_path, upscale_paths}` from `GET /wait` | Tag fires before the response is returned. |
| `JOB_FAILED` | HTTP response: `{status: "failed", error, error_code}` from `GET /wait` | Stable `error_code` mapped to remediation. |
| `MJ_DERIVED_RECEIVED` | PNG/WebP at `{output_dir}/{asset_id}_{kind}_{uuid8}.{ext}` | A vary/zoom/pan/upscale/animation result from an `mj_action` press. |
| `ASSET_PROMOTED` | PNG file at consumer's `dest` path | Idempotent — overwrites. |
| `PROMPT_LOGGED` | JSONL line in `{prompt_log_path}` | Append-only. |

## Design rules

- **Past-tense names.** Tags are `{ENTITY|SUBSYSTEM}_{VERB_PAST_PARTICIPLE}`: `RECEIVED` not `RECEIVE`, `VALIDATED` not `VALIDATE`, `COMPLETED` not `COMPLETE`. The same event always carries the same identifier.
- **`has_*` fields carry shape, not data.** A boolean named `has_*` records that something is present without recording its value — `CONFIG_VALIDATED.has_guild_id`, `PROMPT_LOGGED.has_job_id` / `has_error`. A trace shows that a guild was configured, not which one.
- **Branchable values are locked enums.** Every field a client might branch on — `JOB_FAILED.error_code`, `CONFIG_VALIDATION_FAILED.code`, `GRID_MATCHED.match_path` — is constrained to a fixed set and enforced at emit, so a typo'd or out-of-contract value fails the call instead of reaching a consumer.
- **Growth is additive.** New events are appended; existing ones are never renamed, removed, or repayloaded within 0.1.

## Growing the catalog

`0.1` is locked: the existing tags are frozen — no renames, no removals, no payload changes to an event that has shipped. The catalog still grows within 0.1 by adding new events; only a change that would break a consumer reading older streams (a rename, a removal, a payload change) moves to 0.2, where removals become deprecation entries rather than deletes. A future backend's events arrive this way.

To add an event:

1. Add the entry to `vocabulary/0.1.json` **and** the package-data copy at `packages/python/src/cascade_img/vocabulary/versions/0.1.json` (kept identical; the checks read the package copy), and update `tag_set_hash` in the same commit.
2. Regenerate the reference: `python3 packages/python/tools/render_vocabulary_reference.py`.
3. Then write the `emit()` callsite. The parity check fails on an emit of an undeclared tag, and the reference check fails if `0.1-reference.md` is stale.
