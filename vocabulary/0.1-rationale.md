# vocabulary/0.1-rationale.md — cascade-img vocabulary v0.1

*The companion rationale to `vocabulary/0.1.json`. v0.1 locked 2026-06-02.*

---

## Intent and scope

This vocabulary names the events cascade-img emits at runtime — what the bridge daemon does, what the MCP server does, what the CLI does, what the curation primitives do.

We have found that doing so makes LLM-assisted development easier, because the code is designed around a fixed event grammar rather than ad-hoc logging. Because every runtime action maps to a named event with a typed payload, a reader or an agent can reconstruct what the program did from the event stream alone, instead of parsing free-text prose logs.

In scope: the v0.1 surface (MJ Discord backend + composer + curation + log + MCP + CLI). Out of scope: future backends (Flux, DALL-E, Imagen) — those bring their own backend-specific tags at v0.2+.

---

## Per-layer decisions

### Layer 0 — Ontology (entities)

Names the durable things the system tracks state for, so every tag's foreign keys resolve to a known entity.

Implicitly defined by the v0.1 tag set. The entities cascade-img tracks state for are:

- **Job** — one /imagine roll through to its terminal state (done | failed). Identified by `job_id`. Most tags carry `job_id` + `asset_id` foreign keys to this entity.
- **Asset** — a named output the user is generating (e.g. `"mountain-icon"`, `"hero-portrait"`). Identified by `asset_id`. Carried in every job-category tag's payload.
- **Bridge** — the long-running Flask + discord.py-self daemon process. Operated through HTTP. Most config-category and bridge-category tags concern it.
- **Backend** — the pluggable backend implementing `ImageGenerationBackend`. v0.1 has one (`MidjourneyDiscordBackend`). The `backend` payload field on `CASCADE_INIT` carries which is active.

Layer 0 is not yet structured as an explicit `ontology.entities[]` array in the JSON; the entities are implicit in the foreign-key fields. Deferred to v0.2 if/when a second backend forces explicit cross-backend entity reconciliation.

### Layer 1 — Lexical (tag names)

Fixes the spelling and naming form of every tag, so the same event always carries the same identifier.

27 tags. Names follow the `{ENTITY|SUBSYSTEM}_{VERB_PAST_PARTICIPLE}` form. Examples:

- `CASCADE_INIT` (session start)
- `CONFIG_VALIDATED` / `CONFIG_VALIDATION_FAILED`
- `DISCORD_CONNECTED`
- `BRIDGE_HEALTHY` / `BRIDGE_CHECKENV_RAN` / `BRIDGE_DOCTOR_RAN`
- `IMAGINE_FIRED`, `GRID_MATCHED`, `GRID_RECEIVED`, `UPSCALE_REQUESTED`, `UPSCALE_RECEIVED`, `JOB_COMPLETED`, `JOB_FAILED`
- `BACKEND_HTTP_CALLED`
- `CROP_QUADRANT`, `ALPHA_KEY_APPLIED`, `ASSET_PROMOTED`
- `PROMPT_COMPOSED`, `PROMPT_LOGGED`
- `MCP_SERVER_STARTED`, `MCP_TOOL_CALLED`, `MCP_TOOL_COMPLETED`, `MCP_TOOL_FAILED`
- `CLI_ROLL_STARTED`, `CLI_ROLL_COMPLETED`, `CLI_ROLL_FAILED`

The imagine-to-promote loop fires as `IMAGINE_FIRED → GRID_MATCHED → GRID_RECEIVED → (UPSCALE_REQUESTED → UPSCALE_RECEIVED)* → JOB_COMPLETED`.

Past-tense forms enforced: `RECEIVED` not `RECEIVE`; `MATCHED` not `MATCH`; `VALIDATED` not `VALIDATE`. Irregular forms handled explicitly (`COMPLETED` not `COMPLETE` to match past-tense convention).

### Layer 2 — Payload (typed fields per tag)

Declares the typed fields each tag carries, so every event has a known shape rather than an open bag of data.

Required fields named per tag; optional fields are an empty array at v0.1 (none of the tags carry optional payloads yet). Foreign keys (`asset_id`, `job_id`) resolve to the Layer 0 entities above.

The `match_path` field on `GRID_MATCHED` carries the value `"pending"` (first-touch match) or `"progress_fallback"` (the V7 patch path), recording which match path fired.

The `has_*` fields (e.g. `has_guild_id`, `has_job_id`, `has_error`) carry boolean shape information without leaking the underlying values. A missing guild_id surfaces via `CONFIG_VALIDATED.has_guild_id=false` rather than as a free-text warning.

### Layer 3 — Session (event stream framing)

Marks where an event stream begins and ends, so individual events can be grouped into one process or tool session.

Implicit at v0.1. `CASCADE_INIT` is the session-boundary-open tag for the daemon process; there is no explicit boundary-close tag (the process either runs until killed, or `CONFIG_VALIDATION_FAILED` exits before the session formally begins). MCP server sessions framed by `MCP_SERVER_STARTED → MCP_TOOL_CALLED* → ...` with no explicit close.

Deferred to v0.2 — add `BRIDGE_SHUTDOWN` and `MCP_SERVER_STOPPED` as boundary-close tags.

### Layer 4 — Temporal (timing invariants)

Records the expected timing between events, so a stalled or slow run can be distinguished from a normal one.

Implicit at v0.1. Known cadence:

- `IMAGINE_FIRED` followed by `GRID_MATCHED` within roughly 30-90s (grid mode), 2-4 min (single upscale), 5-8 min (`upscale=all`).
- `BRIDGE_HEALTHY` fires on each `/health` call; no fixed cadence — driven by the client's polling.
- `MCP_TOOL_CALLED` paired with `MCP_TOOL_COMPLETED` or `MCP_TOOL_FAILED` within tool-specific timeouts.

Deferred to v0.2: encode the timeout bands as explicit temporal invariants.

### Layer 5 — State-Transition (ordering graph)

Defines which events must precede or follow which others, so out-of-order sequences can be flagged as invalid.

Implicit at v0.1. Known invariants from the daemon's behavior:

- `CONFIG_VALIDATED` must fire before any other session-category tag (or `CONFIG_VALIDATION_FAILED` and the process exits).
- `DISCORD_CONNECTED` must fire before any `IMAGINE_FIRED` (gated by `_ready.is_set()`).
- `GRID_MATCHED` must precede `GRID_RECEIVED` for the same `job_id`.
- `UPSCALE_REQUESTED.slot=N` must precede `UPSCALE_RECEIVED.slot=N` for the same `job_id`.
- `JOB_COMPLETED` and `JOB_FAILED` are terminal for a given `job_id` — no subsequent tags carry that `job_id`.

Deferred to v0.2 as explicit `state_transitions[]` array entries in the JSON.

### Layer 6 — Runtime / Operator

Identifies which component emits which tags, so every event can be traced back to the part of the system that produced it.

The named operators in v0.1:

- **`MidjourneyDiscordBackend`** — emits `BACKEND_HTTP_CALLED`. Consumes `imagine`, `wait`, `status`, `health` calls from upstream consumers.
- **Bridge daemon** (Flask + discord.py-self) — emits `CASCADE_INIT`, `CONFIG_*`, `DISCORD_CONNECTED`, `BRIDGE_HEALTHY`, `IMAGINE_FIRED`, `GRID_*`, `UPSCALE_*`, `JOB_*`.
- **`PromptComposer`** — emits `PROMPT_COMPOSED`.
- **`PromptLog`** — emits `PROMPT_LOGGED`.
- **Curation primitives** — emit `CROP_QUADRANT`, `ALPHA_KEY_APPLIED`, `ASSET_PROMOTED`.
- **MCP server** — emits `MCP_SERVER_STARTED`, `MCP_TOOL_CALLED`, `MCP_TOOL_COMPLETED`, `MCP_TOOL_FAILED`.
- **`cascade-mj` CLI** — emits `CLI_ROLL_STARTED`, `CLI_ROLL_COMPLETED`, `CLI_ROLL_FAILED`.

Deferred to v0.2 as explicit `operators[]` array.

### Layer 7 — Evidence (payload constraints beyond type signatures)

Pins specific payload fields to a fixed set of allowed values, so any value outside that set is detectable as drift.

- `JOB_FAILED.error_code` is constrained to a stable set: `DISCORD_400_OUTDATED`, `DISCORD_400_UNKNOWN_CHANNEL`, `DISCORD_401`, `GRID_DOWNLOAD_FAILED`, `MJ_UUID_MISSING`, `SUBMIT_FAILED`, `UPSCALE_DOWNLOAD_FAILED`. Anything else is drift.
- `CONFIG_VALIDATION_FAILED.code` is constrained to: `MISSING_DISCORD_TOKEN`, `MISSING_CHANNEL_ID`, `INVALID_CHANNEL_ID`, `MISSING_IMAGINE_VERSION`, `INVALID_PORT`. Anything else is drift.
- `MCP_TOOL_*.tool` is constrained to the 10 tool names in the MCP server module.
- `GRID_MATCHED.match_path` is constrained to `"pending"` or `"progress_fallback"`.

These constraints are documented in each tag's `note` field; runtime enforcement of enum value membership is deferred to v0.2 (currently emit validates tag existence + required-field presence, not enum values).

### Layers 8-10 — Report / Version / Grammar-Growth

Cover how events roll up into reports, how the vocabulary is versioned and locked, and how it is allowed to grow over time.

- **Layer 8 (Report binding):** Behavior tags → OBSERVED/EXPECTED; incident tags → NOTES; proposals → GRAMMAR_DRIFT. No view tags in v0.1.
- **Layer 9 (Version):** `0.1`, locked 2026-06-02, locked_by "Green Rose Systems via cascade-img v0.1.0a1 ship". prior_version: null.
- **Layer 10 (Grammar-Growth):** Validator-extras strict at the package; production may relax via `CASCADE_STRICT_SIGNALS=false` env var.

## Tag-to-artifact table

cascade-img has no UI; its observable outputs are FILE OUTPUTS (PNGs written to disk) and HTTP RESPONSES (structured JSON returned to clients). This table pairs each behavior tag with the artifact it corresponds to:

| Behavior tag | Artifact counterpart | Notes |
|---|---|---|
| `IMAGINE_FIRED` | HTTP response: `{job_id, asset_id, status: "submitted", upscale}` from `POST /imagine` | Tag fires after the response is returned. |
| `GRID_MATCHED` | None at the consumer surface — internal state transition | Visible only in the event stream. |
| `GRID_RECEIVED` | PNG file at `{output_dir}/{asset_id}.{png|webp}` | The `path` and `bytes` payload fields identify the artifact. |
| `UPSCALE_RECEIVED` | PNG file at `{output_dir}/{asset_id}_u{slot}.{png|webp}` or `.png` | Same shape. |
| `JOB_COMPLETED` | HTTP response: `{status: "done", image_path, grid_path, upscale_paths}` from `GET /wait` | Tag fires before the response is returned. |
| `JOB_FAILED` | HTTP response: `{status: "failed", error, error_code}` from `GET /wait` | Stable `error_code` mapped to remediation. |
| `ASSET_PROMOTED` | PNG file at consumer's `dest` path | Idempotent — overwrites. |
| `PROMPT_LOGGED` | JSONL line in `{prompt_log_path}` | Append-only. |

## Project-specific notes

- **Strict validation:** Payload fields not declared in the schema raise at emit. The `CASCADE_STRICT_SIGNALS=false` env var relaxes this in production.
- **Vocabulary location:** Canonical at project root `vocabulary/0.1.json` AND bundled in the package at `packages/python/src/cascade_img/vocabulary/versions/0.1.json` (the package-data copy for the wheel). The two are kept identical; the CI parity step reads the package-data copy.
