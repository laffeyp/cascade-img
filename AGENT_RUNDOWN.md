# Agent Rundown

For an agent (or a person) being onboarded to this repository. Open it in any
LLM (Claude, etc.) and paste the prompt below. It
will read the source of truth and give you an accurate, grounded rundown of
cascade-img — what it is, how it works, and how to run it.

The prompt is written to make the model **read the current files and
synthesize**, not recall or guess, so it stays accurate as the code evolves. It
is told to flag anything it can't verify.

---

```text
You are onboarding me to the `cascade-img` repository that you have open. Be
accurate and concrete. Ground every claim in the files named
below, quoting a path (and line or tag name) when you assert something
important. Do not invent capabilities; if something is not verifiable from
these files, say so.

Read these in full, in order:
  1. README.md                 — what it is and the headline.
  2. ARCHITECTURE.md           — components, data flow (CLI/MCP -> HTTP bridge ->
                                 backend -> Discord -> Midjourney), the response
                                 envelope, and the job state machine.
  3. AGENTS.md                 — how an LLM is meant to drive it: the
                                 compose -> imagine -> wait -> curate -> log loop.
  4. RUNBOOK.md                — install, credential capture, bring-up, and how
                                 to recover from each failure mode.
  5. packages/python/src/cascade_img/vocabulary/versions/0.1.json
                               — the catalog of events the program reports, with
                                 payloads and notes. Read the JOB_FAILED entry
                                 for the stable error codes.
  6. packages/python/pyproject.toml  — the real console-script names under
                                 [project.scripts]; quote them exactly.
  7. CHANGELOG.md             — current state, known limits, and the
                                 Discord-user-account Terms-of-Service reality.

Then give me, in plain language and in this order:

  - WHAT IT IS, in two sentences.
  - THE MENTAL MODEL: composable prompt parts -> compose -> imagine -> wait ->
    curate (crop / alpha-key / promote) -> log; and the split between the
    long-running bridge daemon and the CLI/MCP entry points.
  - HOW TO RUN IT end to end: the three operating console scripts (plus
    `cascade-trace-check`, a trace validator; names taken verbatim from
    [project.scripts]), the required .env values (from RUNBOOK.md), and the
    minimal happy path from install to a first generated image.
  - THE CONTRACT: the { ok, result | error: { code, message, remediation } }
    envelope, and what the stable error codes mean (from the JOB_FAILED tag).
  - THE CAVEATS: the Discord-user-account / Terms-of-Service reality
    (from README.md / RUNBOOK.md) and the known limits (from CHANGELOG.md /
    ARCHITECTURE.md).
  - WHERE TO GO NEXT: ask me what I'm trying to do, then point me at the exact
    file and section.

Rules:
  - If a prose doc and the code disagree, trust the code and flag the drift
    explicitly.
  - Keep it tight. I want the rundown, not a recital. Use short sections.
  - Do not add any AI-attribution or "generated by" lines anywhere.
```

---

Why this exists: cascade-img is built to be operated by an LLM, so the fastest
way in is to let an LLM read its own source and explain it back to you. If you
only read one human-facing file first, read [ARCHITECTURE.md](./ARCHITECTURE.md);
if you're about to run it, read [RUNBOOK.md](./RUNBOOK.md).
